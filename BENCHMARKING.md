# Benchmarking Guide

## Overview

This project uses benchmarks to track performance at all stages of development. It is especially important to monitor performance when introducing new abstraction layers, such as adapters for different loggers.

## Benchmark Files

The project maintains several benchmark files:

- **[`benchmarks_baseline.txt`](./benchmarks_baseline.txt)** - Baseline performance metrics before making changes. Create with `make bench-baseline`.
- **`benchmarks.txt`** - Current benchmark results. Generated by `make bench`.
- **[`benchmarks_compare.txt`](./benchmarks_compare.txt)** - Statistical comparison between baseline and current results. Generated by `make bench-compare`.

The comparison file shows:
- Performance changes (faster/slower)
- Memory usage differences
- Allocation count changes
- Statistical significance indicators

### Key Performance Metrics

Current performance after adapter implementation and optimization:

- **Direct zap**: 0.34 ns/op, 0 allocs
- **xlog wrapper**: 2.04 ns/op, 0 allocs (overhead ~1.7 ns)
- **Context reuse**: 1.95-2.12 ns/op (0 allocs) vs 62-142 ns/op when created in a loop
- **Field conversion to OTel**: 85-295 ns/op with 1 alloc (optimized, was 2-3x slower before)

## Running Benchmarks

### Quick Start with Make Commands

The easiest way to run and compare benchmarks is using the provided Make targets:

```bash
# Show all available benchmark commands
make bench-help

# Run benchmarks and save to benchmarks.txt
make bench

# Save current benchmarks as baseline
make bench-baseline

# Compare current benchmarks with baseline
make bench-compare

# Run benchmarks and compare with baseline in one command
make bench-full

# Run quick benchmarks (1s per test instead of 3s)
make bench-quick

# Run benchmarks with CPU profiling
make bench-cpu

# Run benchmarks with memory profiling
make bench-mem
```

### Custom File Names

You can specify custom file names for benchmarks:

```bash
# Save benchmarks to a custom file
make bench BENCHMARK_FILE=after_optimization.txt

# Create a custom baseline file
make bench-baseline BASELINE_FILE=before_adapters.txt

# Compare specific files
make bench-compare BASELINE_FILE=before.txt BENCHMARK_FILE=after.txt
```

### Manual Benchmark Execution

If you prefer to run benchmarks manually:

```bash
# Run all benchmarks with 3-second execution time
go test -bench=. -benchmem -benchtime=3s -run='^$'

# Basic loggers only
go test -bench=BenchmarkLogger -benchmem -benchtime=3s

# Context operations only
go test -bench=BenchmarkAdvance -benchmem -benchtime=3s

# Tracing only
go test -bench=BenchmarkWithOperationSpan -benchmem -benchtime=3s

# Save results to a file for later comparison
go test -bench=. -benchmem -benchtime=3s -run='^$' | tee my_benchmarks.txt

# Run with multiple iterations for statistical confidence
go test -bench=. -benchmem -benchtime=3s -count=10 -run='^$' | tee benchmarks_reliable.txt
```

## Performance Comparison

### Using Make (Recommended)

The simplest way to compare benchmarks:

```bash
# 1. Create baseline (do this before making changes)
make bench-baseline

# 2. Make your changes to the code

# 3. Run benchmarks and compare
make bench-full
```

### Manual Comparison with benchstat

```bash
# Install benchstat (if not installed)
go install golang.org/x/perf/cmd/benchstat@latest

# Compare two files
benchstat benchmarks_baseline.txt benchmarks.txt
```

### Interpreting benchstat Results

The comparison output (`benchmarks_compare.txt`) shows three tables:

#### 1. Performance (sec/op)
```
name                    old time/op  new time/op  delta
Logger/zap-14           0.35ns ± 0%  0.37ns ± 0%  +5.71%
Logger/xlog-14          2.33ns ± 0%  2.98ns ± 0%  +27.90%
```

#### 2. Memory Usage (B/op)
```
name                    old B/op     new B/op     delta
Logger/zap-14           0.00 ± 0%    0.00 ± 0%    ~
```

#### 3. Allocations (allocs/op)
```
name                    old allocs   new allocs   delta
Logger/zap-14           0.00 ± 0%    0.00 ± 0%    ~
```

**Reading the results:**
- `~` (tilde) - no statistically significant difference
- `+X%` - performance **degraded** by X% (slower/more memory/more allocs)
- `-X%` - performance **improved** by X% (faster/less memory/fewer allocs)
- `± Y%` - measurement error margin
- `geomean` - geometric mean of all benchmarks (overall trend)

**Statistical notes:**
- `¹ need >= 6 samples for confidence interval` - run more iterations for accuracy
- `² all samples are equal` - no variance detected
- `³ need >= 4 samples to detect a difference` - insufficient data for statistical significance
- `(p=1.000 n=1)` - only 1 sample, not statistically reliable

**Recommendation:** For reliable results, run benchmarks with `-count=10` to get statistical confidence.

### Analyzing the Comparison File

When reviewing `benchmarks_compare.txt`, focus on:

1. **Geomean row** (last row) - overall performance trend:
   - Look at sec/op, B/op, and allocs/op columns
   - Small changes (±5%) are usually acceptable
   - Large changes (>10%) require investigation

2. **Critical paths** (operations that run frequently):
   - Basic logging operations (Logger/*)
   - Context reuse patterns (reuse_*)
   - Field conversion (ConvertFieldsToAttributes/*)

3. **Allocation increases** - most critical indicator:
   - New allocations in hot paths are concerning
   - Even small allocation increases can impact GC pressure

4. **Memory usage changes**:
   - Consistent with allocation changes
   - Check if memory grows proportionally to new allocations

**Example analysis workflow:**
```bash
# 1. Generate comparison
make bench-compare > analysis.txt

# 2. Check geomean
tail -3 benchmarks_compare.txt

# 3. Look for significant regressions (>10%)
grep "+[0-9][0-9]\." benchmarks_compare.txt

# 4. Check for new allocations in critical paths
grep "allocs/op" benchmarks_compare.txt | grep "Logger\|reuse"
```

## Acceptance Criteria After Adapter Implementation

After implementing the adapter layer we expect:

### ✅ Acceptable

- Basic logging: **< 5 ns/op** additional overhead
- Context operations: **< 10 ns/op** additional overhead
- Memory allocations: **no changes**
- Field conversion: **< 10% slowdown**

### ⚠️ Requires Attention

- Basic logging: **5-10 ns/op** overhead
- Context operations: **10-20 ns/op** overhead
- Additional allocations: **+1 alloc/op**

### ❌ Unacceptable

- Basic logging: **> 10 ns/op** overhead
- Context operations: **> 20 ns/op** overhead
- Significant allocation increase: **+2 or more alloc/op**

## Common Bottlenecks

### 1. Interface Dispatch Overhead

```go
// Problem: too many abstraction levels
type Logger interface {
    Log(msg string)
}

type Adapter struct {
    logger Logger  // another level
}

// Solution: minimize nesting levels
```

### 2. Allocation in Hot Path

```go
// Problem: creating slices on every call
func (a *Adapter) Log(msg string, fields ...Field) {
    attrs := make([]slog.Attr, len(fields))  // allocation!
    // ...
}

// Solution: reuse or pre-allocate
```

### 3. Excessive Type Conversion

```go
// Problem: multiple conversions
func convert(f Field) SlogAttr {
    switch f.Type {
    case IntType:
        return slog.Int64(f.Key, int64(f.Value))  // conversion
    }
}

// Solution: store in optimal format from the start
```

## Profiling

### CPU Profiling

```bash
# Create CPU profile
go test -bench=BenchmarkLogger -cpuprofile=cpu.prof

# Analyze profile
go tool pprof -http=:8080 cpu.prof
```

### Memory Profiling

```bash
# Create memory profile
go test -bench=BenchmarkLogger -memprofile=mem.prof

# Analyze profile
go tool pprof -http=:8080 mem.prof
```

### Trace Profiling

```bash
# Create trace
go test -bench=BenchmarkLogger -trace=trace.out

# Analyze trace
go tool trace trace.out
```

## Continuous Monitoring

### In CI/CD

Add performance checks to CI:

```yaml
# Example for GitHub Actions
- name: Run benchmarks
  run: |
    go test -bench=. -benchmem -benchtime=1s -run='^$' | tee new.txt
    benchstat benchmarks_baseline_before_adapters.txt new.txt
```

### Local Development

Before each major change:

```bash
# 1. Record current state
go test -bench=. -benchmem -benchtime=3s -run='^$' > before.txt

# 2. Make changes

# 3. Compare
go test -bench=. -benchmem -benchtime=3s -run='^$' > after.txt
benchstat before.txt after.txt
```
